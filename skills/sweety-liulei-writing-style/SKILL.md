# 刘磊写作风格

这个 skill 负责“像用户本人”的部分：读取全局风格库、检索相似样本、记录人工改稿差异，并把偏好转成下一次写作可执行的约束。

它不替代 `sweety-humanize-writing`。两者关系：

- `sweety-humanize-writing`：通用去 AI 味、删套话、保事实、改节奏。
- `sweety-liulei-writing-style`：个人口吻、取舍、标题、开头、段落收束、样本学习。

默认组合顺序：

```text
真实素材/亲笔碎片/初稿 -> sweety-humanize-writing 做底层清洁 -> 本 skill 套用个人 profile -> 人工编辑器确认 -> 写入样本库
```

## 全局风格库

默认根目录：

```bash
${SWEETY_WRITING_STYLE_HOME:-$HOME/.codex/writing-style/liulei}
```

建议结构：

```text
profiles/
  global/profile.md
  wechat-deep-article/profile.md
  group-infographic/profile.md
  technical-explanation/profile.md
samples/
  <profile>/<sample-id>/
    ai-draft.md
    human-final.md
    diff-notes.md
```

写作前按顺序读取：

1. `profiles/global/profile.md`
2. 与当前任务最接近的子 profile，例如 `profiles/wechat-deep-article/profile.md`
3. `samples/<profile>/` 中 1-3 个最相似样本的 `diff-notes.md`，必要时再读 `ai-draft.md` 和 `human-final.md`

如果用户提到朱雀、Matrix、AI 检测、AI 率、人工创作特征，或某篇稿件被检测为高 AI 率，必须额外读取 `references/zhuque-calibration.md`。

如果用户是因为检测结果要求返修，先判断是否有足够真实素材和人工主体文本。没有素材时，不直接生成“检测级终稿”，而是输出待补素材问题或摘编方案。

## Profile 选择

| 场景 | profile |
|---|---|
| 公众号深度文章、视频整理、国漫长文 | `wechat-deep-article` |
| 掘金技术文章、AI/Agent 观点、前端工程方案 | `juejin-technical-writing` |
| 群聊讲解图、图解卡片、观点拆解图 | `group-infographic` |
| 技术方案、架构说明、工具实现复盘 | `technical-explanation` |
| 场景不明确 | 先读 `global`，再根据任务自行判断 |

不要把一个场景的风格硬迁移到另一个场景。公众号口吻不能污染技术文档，技术说明也不能写成公众号爆文。

## 核心偏好

读取 profile 后，优先应用这些稳定偏好：

- 先给判断，再给材料，不要先写长铺垫。
- 不要把每段都写成完整闭环；允许“判断 + 两三个例子 + 收住”。
- 删除定义式、解释式、重复式句子，尤其是“观点 + 解释 + 再解释”。
- 保留一点真实口语和不规整语序，但不学习错字、病句和事实错误。
- 标题可以有主观判断，但必须有对象、问题和点击理由。
- 视频类深文优先顺原材料精炼，不抢原作者观点；如果人工样本显示可转第一人称，再转成“我的判断 + 材料支撑”。
- 技术观点类文章优先学习 `juejin-technical-writing`：具体问题开场，短段推进，个人判断接工程解释，再给可执行路线。
- 任何新增事实、数据、经历、引用都必须来自用户材料或可核验来源。

## 真实口吻与 AI 腔清理

用户新增的稳定偏好：改稿时先识别并消灭 AI 腔，再谈润色。凡是看起来太对、太顺、太完整的句子，都要怀疑它是不是机器在替作者说话。

逐段检查这四个死亡信号：

1. 过渡词堆积：看到“首先、其次、最后、综上所述、不仅如此、值得注意的是”等机械连接词，默认删掉。上下文能读懂的地方不要补连接词，确实需要承接时只用“但、所以、还有、重点是、说白了”等短词。
2. 句子太整齐：连续三句长度、结构、语气都差不多时，必须打破。可以一句很短，也可以把两句合成长句；重点是让段落像真实思考，而不是像提纲排队。
3. 情绪太正向：不要把挫折写成“但是，正是这段经历让我……”。真实作者可以停在困难里，先把难处讲清楚，不急着给正能量和标准解法。
4. 举例太知名：不要默认使用乔布斯、马斯克、比尔盖茨这类公共名人。能用用户自己的真实经历，就用用户经历；能用身边普通人的可匿名事实，就用普通人场景；没有材料时，宁可标注“待补真实经历”，不能编。

个人经历规则：

- 公众号深文、职业文章和观点稿，优先把抽象案例换成用户自己的真实故事、项目过程、工具页面、报错、选择、犹豫和复盘。
- 如果用户没有提供经历，只能写“我在整理这篇资料时看到/意识到/卡住的点”这类真实生产过程，或输出待补素材问题。
- 禁止伪造用户的求职、裁员、收入、公司、家庭、病痛、关系和具体项目经历。

语气规则：

- 把“正确但不像人说话”的句子改成用户平时会说的判断句。
- 少写标准答案，多写真实取舍：哪里不确定、哪里只能先做低成本动作、哪里需要读者自己核对。
- 允许一点粗粝感：短句、停顿、追问、半截判断都可以，但不能变成错字、病句和事实含混。

排版规则：

- 小标题要帮读者扫读，优先使用问题、动作或具体对象，不写抽象目录。
- 表情包、配图和梗图只作为公众号排版阶段的“配图位”记录，不在正文阶段机械插入符号或表情。
- 长文每 300-500 字至少有一次扫读停顿：小标题、加粗判断、短列表或引用资料说明。

## 共鸣与气口

用户新增的稳定偏好：文章要能调动读者共鸣，不能写成十全十美的满分答案。作者是人，就要允许露出漏洞、限制、取舍和不确定；也可以给一个足够明确、可被讨论的强判断。读者有地方补充、反驳、纠正，才更愿意输出观点。

执行规则：

- 每篇公众号观点稿至少留一个“气口”：一个真实缺点、一个未覆盖场景、一个有争议判断、一个待补经历、一个取舍边界，或一个明确邀请读者纠正的问题。
- 气口必须是真实边界，不是故意写错事实、乱下结论或制造逻辑漏洞。
- 不要把文章写成“我都想明白了，你照做就行”。更像本人的是：“我现在只能先判断到这里，你可以不同意，但我会先这么做。”
- 强判断可以出现，但后面要有证据和边界。例如“先别学工具”可以写，但要说明适用场景，不要写成所有人都该这么做。
- 结尾不要只写客服腔的欢迎留言。优先问一个能让读者补充经验的问题，例如“你现在卡住的是哪一类任务？”“这张表你会怎么改？”“我漏掉了哪种情况？”

## 基本改稿要求

用户的飞书文档《8个【万能改稿】提示词》是本 skill 的基础改稿动作库。使用时先判断当前稿件真正需要哪几项，不要八项机械全跑。

默认改稿顺序：

1. 逻辑优化：先看文章结构、论证薄弱处、因果跳跃和过渡生硬处。保持原观点不变，只修推理链和材料顺序。
2. 整体润色：修正语病、别扭表达、重复口头禅和不自然转折，让句子更顺，但不擅自增删主要段落。
3. 节奏调整：调整长短句、段落停顿和设问位置，让文章有呼吸感。短句只能用来停顿或加重判断，不能堆成伪金句。
4. 情感增强：补共情、场景、个人感受或细节时，只能来自原材料、用户经历或可核验信息。禁止为了“有温度”编故事。
5. 精简压缩：当稿件解释欲过强时，默认压到原文 60%-70% 左右；删除冗余表达、重复论述、弱相关内容，保留核心观点和关键论据。
6. 扩写丰富：当稿件太薄时，再补案例、数据、细节或多角度分析；扩写幅度通常不超过 1.5-2 倍，新增内容必须有来源。
7. 风格转换：可以转换正式、口语、故事、调侃、激昂等表达方式，但事实、核心观点和用户个人 profile 优先级高于目标风格标签。
8. 传播优化：标题、开头、小标题要自然埋入对象、问题、关键词和点击理由；公众号场景优先点击、完读和评论，不做关键词堆砌。

执行边界：

- 改稿不是重写。除非用户明确要求，默认保留原材料的观点、顺序和信息边界。
- 先修逻辑和信息密度，再修文采；不要先把文字润得漂亮。
- 示例里出现的数据、经历、引用只代表写法，不代表可以照搬或编造。
- 对用户说“精简”“顺着逐字稿”“不要自己改写”时，优先使用逻辑优化、精简压缩和节奏调整，谨慎使用情感增强和扩写丰富。

## 人工质感验收

用户飞书文档《如何让AI文章100%通过朱雀AI检测》只沉淀为写作质量验收，不沉淀为规避检测技巧。目标是让文章更像真实作者认真写过，而不是故意制造错误。

如果朱雀返回 100% AI 或高 AI 率，先按 `references/zhuque-calibration.md` 做失败复盘。默认根因不是连接词或标点，而是缺少真实素材层、观点链太完整、伪第一人称和安全结论过多。

改稿后必须检查：

- 连接词密度：删掉“然而、因此、不仅如此、综合而言、值得注意的是”等机械连接。能靠上下文读懂的地方，直接删；需要保留时换成“但、所以、还有、重点是、说白了”等朴素表达。
- 标点自然度：标点服务语气和停顿，不要每段都同一种节奏。禁止故意混用半角/全角标点、错别字、异常空格来伪装人工痕迹。
- 句式完整度：避免每句都是标准“主语 + 谓语 + 宾语”。可以有短句、问句、省略句和单独成行的判断，但必须符合中文阅读习惯。
- 用词朴素度：删除“卓越、杰出、显著、深度、全面、赋能、生态、体系化”等华丽或公文词。优先用更短、更具体、更接近说话的词。
- 段落不规整：避免每段长度接近、结构对称、像提纲排队。允许长短段交错，但每一段都必须有信息推进。
- 自读门禁：最终稿必须自己读一遍。凡是读起来别扭、不像用户平时会说的话、像教程腔或营销腔的句子，重写。
- 真实素材门禁：开头 200 字内必须有具体工具、项目、页面、报错、会议、代码、时间点或作者经历；全文至少 5 个事实锚点。没有这些锚点，不得作为终稿或提交朱雀。

硬边界：

- 不承诺通过任何 AI 检测，也不以“降低 AI 率”为直接写作目标。
- 不使用“故意多犯错误”“错标点”“异常空格”“降 AI 工具”等伪人工技巧。
- 如果检测压力和读者体验冲突，优先读者体验、事实可靠性和用户真实表达。
- 缺真实经历或具体材料的题目，不能直接生成检测级终稿；应输出待补素材草稿或先问 3-5 个素材问题。
- 朱雀检测级终稿不得由 AI 从零生成完整文章；必须以用户亲笔文本或用户人工编辑版为主体，AI 只做选题、摘编、轻微整理和门禁。
- 不把“同义词替换、错别字、乱标点、硬塞口语词”当作个人风格学习；这些是检测焦虑下的噪声，不进入 profile。

## 通用编辑器

需要让用户人工修改并记录风格时，优先使用脚本：

```bash
node ~/.codex/skills/sweety-liulei-writing-style/scripts/style-editor.mjs \
  --draft /absolute/path/article.md \
  --profile wechat-deep-article \
  --port 4178
```

也可以指定样本 id：

```bash
node ~/.codex/skills/sweety-liulei-writing-style/scripts/style-editor.mjs \
  --draft /absolute/path/article.md \
  --profile wechat-deep-article \
  --sample-id 20260602-example \
  --port 4178
```

确认后脚本会：

- 保留 `article.ai-backup.md`
- 写入 `article.human.md`
- 用人工版覆盖原 `article.md`
- 写入全局样本库 `samples/<profile>/<sample-id>/`
- 追加子 profile 的学习记录

确认后还必须做一次“文风深学习”，不能只停在自动 diff：

1. 读取本次 `samples/<profile>/<sample-id>/diff-notes.md`、`ai-draft.md` 和 `human-final.md`。
2. 从人工稿中提炼可复用的文风规则：标题发问方式、开头身份锚点、时间线写法、口语承接词、判断句式、段落长度、资料嵌入方式、建议组织方式和结尾收束方式。
3. 把这些规则追加到对应 `profiles/<profile>/profile.md`，标题格式为 `## <date> <sample-id> 文风深学习`。
4. 只记录表达策略和删改偏好，不把某次文章的事实当成跨题材可复用事实。涉及岗位、年限、工具、公司、项目等，只能作为“需要真实锚点”的例子。
5. 下次同类写作前，必须优先读取最近的文风深学习记录，而不是只读取自动提炼规则。

## 学习边界

- 学习表达方式，不学习事实错误、错别字、口误、重复段。
- 学习用户如何删、如何压、如何收，不把人工稿当作可复制素材库。
- 腾讯/朱雀等检测结果只能作为发布门禁记录，不作为“规避检测”的写作目标。
- 检测平台引发的临时改词、乱标点、错别字、降 AI 率技巧，不写入个人风格库。
- 如果用户只要求“诊断”，不要擅自改写或学习。

## 输出

写作或改写完成时，简短说明使用了哪个 profile、读取了哪些样本、哪些风格约束生效。若只是输出正文，遵循用户要求，不额外解释。
