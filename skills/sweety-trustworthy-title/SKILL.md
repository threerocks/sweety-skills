# 可信标题生成

这个 skill 用于写中文内容标题。目标不是制造短期诱点，而是让读者快速知道对象、问题、情绪和点击理由，同时不透支信任。

适用场景：公众号文章标题、小红书图文/滑读卡标题、视频标题、封面主字、社媒贴文标题、作品标题、标题冷审、标题批量候选。

先检查 EXTEND.md，优先级如下：

```bash

test -f "${XDG_CONFIG_HOME:-$HOME/.config}/sweety-skills/sweety-trustworthy-title/EXTEND.md" && echo "xdg"
test -f "$HOME/.sweety-skills/sweety-trustworthy-title/EXTEND.md" && echo "user"
```

## 输入门槛

生成标题前必须先明确这些字段。若缺关键字段，先问最少问题；不要靠脑补补齐。

| 字段 | 必需性 | 说明 |
|------|--------|------|
| 主题/对象 | 必需 | 写谁、写什么事、哪类内容 |
| 核心观点 | 必需 | 标题要承载的判断、问题或冲突 |
| 目标读者 | 必需 | 谁会点、为什么在乎 |
| 内容类型 | 必需 | 文章、图文、滑读卡、视频、封面字、作品名等 |
| 真实支撑 | 必需 | 正文实际能交付的事实、情节、结果、清单或方法 |
| 希望强调 | 可选 | 数字、反差、使用场景、收藏价值、评论钩子 |
| 需要规避 | 可选 | 禁用词、敏感点、已用模板、平台风险 |

## 标题党硬门禁

任何候选标题命中以下问题，必须重写，不得进入最终输出。

| 类型 | 判定 | 改法 |
|------|------|------|
| 震惊体 | 把普通信息写成大事将至 | 降低情绪，写清具体变化或结果 |
| 恐吓体 | 先堆后果，再慢慢吐信息 | 直接说风险对象和可验证后果 |
| 绑架体 | 用必须看、所有人都该、不看就亏命令读者 | 改成读者能自主判断的价值点 |
| 悬空体 | 只写这件事、这个变化、那个人，不写主语和结论 | 补对象、动作和结论 |
| 碰瓷体 | 硬贴热点、名人、平台或概念，正文没有信息增量 | 删除借势名头，回到正文真实内容 |
| 许诺体 | 标题把效果说满，正文只剩空话 | 把绝对收益改成可交付结果 |
| 对立体 | 把正常分歧写成阵营开战或群体冲突 | 改成具体冲突、选择或误解 |
| 伪投稿 | 标题暗示真实粉丝、读者或来信，但正文无来源 | 改成“一个常见问题”或删除第三方包装 |

额外禁用：

- 不用“内幕”“全网都在”“再不看就晚了”“颠覆认知”“封神”“炸裂”等虚高词，除非正文有强证据。
- 不用空泛副词承担点击理由，例如“超级、巨、特别、非常、真的、必须、绝对、最”。保留时必须跟具体对象、动作、数字或结果。
- 不把标题写成口号、命令或谜语。

## 生成流程

1. 先把正文压成一句“对象 + 动作/问题 + 结果/判断”。
2. 列出正文能兑现的 3-5 个真实点击理由：数字、反差、场景、结果、选择、稀缺、误解纠正、故事冲突或收藏用途。
3. 从 [references/title-patterns.md](references/title-patterns.md) 选择 2-4 种标题思路，不要所有候选只换同义词。
4. 生成候选标题：默认 20 个；用户要少量时也先在内部扩散，再筛出指定数量。
5. 对每个候选跑硬门禁，删除标题党、悬空、空泛、错配正文和同质化标题。
6. 最终输出时优先给差异明显的候选；如果用户要“直接给一个”，先给首选标题，再给 1 句选择理由。

## 写法要求

- 默认 14-28 个中文字符；封面主字可更短，深度文章可稍长但不能绕。
- 标题必须让读者 3 秒内看懂主题、情绪方向和点击理由。
- 标题必须和正文强相关，不能预支正文没有交付的结果。
- 多用具体对象、数字、结果词、判断词、动作词。
- 标题之间必须角度不同，不能只是替换同义词。
- 语气自然，像人在认真推荐内容，不像喊口号。
- 有余味可以，但余味来自未说尽的判断，不来自隐瞒主语和结论。

## 输出格式

用户没有指定格式时：

```markdown
1. 标题
2. 标题
3. 标题
...
```

用户要求标题冷审时：

```markdown
## 问题
- [标题]：命中的风险和原因

## 改写
1. 新标题
2. 新标题
```

用户要求“给一个最好的”时：

```markdown
首选：标题
理由：它写清了[对象/动作/结果]，没有预支正文。
```

## 与其他写作 skill 的关系

- 与 `sweety-viral-opening` 同用时：先用本文确定标题，再用 opening skill 写标题后的前三句或首段承接；不要让 opening skill 替代标题判断。
- 与 `sweety-humanize-writing` 同用时：先用本文定标题方向，再用 humanize 清掉口号腔、模板腔和空泛副词。
- 与 `sweety-liulei-writing-style` 同用时：先用本文保证标题清楚可信，再套用用户个人标题偏好。
- 在项目 `AGENTS.md` 明确要求使用本 skill 时，所有标题生成、改写和冷审都必须先加载本 skill。
